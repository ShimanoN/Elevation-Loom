## 2. 開発の基本サイクル

### 2.1 全体の流れ（大きな視点）

```
計画 → 実装 → テスト → レビュー → デプロイ → 改善
  ↑                                          ↓
  └──────────────────────────────────────────┘
              （このサイクルを繰り返す）
```

### 2.2 1つの機能を作る流れ（詳細）

#### ステップ1: 計画（Plan）

**目的**: 何を作るか明確にする

```markdown
チェックリスト:
[ ] 作りたい機能を1行で説明できる
[ ] なぜこの機能が必要か理解している
[ ] どのファイルを変更する必要があるか把握した
[ ] 変更範囲が大きすぎないか確認した（大きい場合は分割）
```

**例**:
- ❌ 悪い例: 「アプリを改善する」（抽象的すぎる）
- ✅ 良い例: 「月次レポートページを追加し、月間の合計獲得標高を表示する」

#### ステップ2: ブランチ作成（Branch）

**目的**: 他の作業と独立して変更を管理する

```bash
# 1. 最新の状態を取得
git fetch origin
git checkout main
git pull origin main

# 2. 新しいブランチを作成
git checkout -b feature/monthly-report

# ブランチ名の規則:
# - feature/〇〇: 新機能
# - fix/〇〇: バグ修正
# - docs/〇〇: ドキュメント更新
# - refactor/〇〇: リファクタリング
```

**なぜブランチを使うのか**:
- 複数の機能を同時並行で作業できる
- 失敗しても元に戻せる
- レビューがしやすい

#### ステップ3: 実装（Implement）

**目的**: コードを書く

```markdown
チェックリスト:
[ ] 小さな単位で変更する（1回の変更は少なく）
[ ] 既存のコードスタイルに合わせる
[ ] コメントは必要最小限（コード自体が読みやすいことが重要）
[ ] console.log()は最終的に削除する（デバッグ用は残さない）
```

**開発中の確認事項**:

```bash
# 定期的にテストを実行
npm test

# コードスタイルをチェック
npm run lint

# 自動修正
npm run lint:fix

# フォーマット
npm run format
```

**小さな単位で保存（ローカルコミット）**:

```bash
# 1. 変更したファイルを確認
git status

# 2. 変更をステージング
git add js/monthly-report.js

# 3. コミット（変更を保存）
git commit -m "feat: add monthly data aggregation function"

# コミットメッセージの規則:
# - feat: 新機能
# - fix: バグ修正
# - docs: ドキュメント
# - test: テスト追加
# - refactor: リファクタリング
# - style: コードスタイル（フォーマット）
```

#### ステップ4: テスト（Test）

**目的**: 作った機能が正しく動くか確認する

```markdown
チェックリスト:
[ ] ユニットテストを書いた（関数単位のテスト）
[ ] テストがすべてパスした
[ ] 手動でも動作確認した（実際にブラウザで確認）
[ ] エッジケースを確認した（空入力、最大値など）
[ ] 既存機能を壊していないか確認した
```

**テストの実行**:

```bash
# すべてのテストを実行
npm test

# 特定のファイルのみテスト
npm test -- monthly-report.test.js

# カバレッジを確認
npm run test:coverage

# E2Eテストを実行
npm run e2e
```

**なぜテストが重要なのか**:
- バグを早期に発見できる
- 後から変更しても安心（壊れたらテストが教えてくれる）
- ドキュメント代わりになる（使い方がわかる）

#### ステップ5: レビュー依頼（Pull Request）

**目的**: 他の人（または将来の自分）にコードを見てもらう

```bash
# 1. リモートにプッシュ
git push origin feature/monthly-report

# 2. GitHubでPull Requestを作成
# （ブラウザでGitHubにアクセス）
```

**Pull Request（PR）の書き方**:

```markdown
## 変更内容

月次レポート機能を追加しました。

## 変更の理由

月間の合計獲得標高を確認したいというニーズがあったため。

## 変更したファイル

- `js/monthly-report.js`: 月次データ集計関数
- `monthly-report.html`: 月次レポートページ
- `test/monthly-report.test.js`: テストコード

## テスト結果

- [x] ユニットテスト: すべてパス
- [x] E2Eテスト: すべてパス
- [x] 手動テスト: 動作確認済み

## スクリーンショット

（月次レポート画面のスクリーンショットを添付）

## レビューポイント

- データ集計ロジックが正しいか確認してください
- UIが直感的か確認してください
```

**レビューで見られるポイント**:
- コードが読みやすいか
- テストがあるか
- 既存のコードを壊していないか
- セキュリティ上の問題がないか

#### ステップ6: 修正（Revision）

**目的**: レビューのフィードバックを反映する

```bash
# 1. フィードバックに基づいて修正
# （コードを編集）

# 2. 再度コミット
git add .
git commit -m "fix: address review comments"

# 3. プッシュ（PRが自動更新される）
git push origin feature/monthly-report
```

**レビューコメントへの対応**:
- ✅ 指摘を素直に受け入れる（学習の機会）
- ✅ 質問があれば遠慮なく聞く
- ✅ 理解できない指摘は理由を聞く

#### ステップ7: マージ（Merge）

**目的**: メインブランチに変更を取り込む

```markdown
チェックリスト:
[ ] すべてのテストがパスしている
[ ] レビュー承認を得た
[ ] コンフリクト（競合）がない
[ ] CI/CDが成功している
```

**マージ後の作業**:

```bash
# 1. メインブランチに戻る
git checkout main

# 2. 最新の状態を取得
git pull origin main

# 3. 作業ブランチを削除（オプション）
git branch -d feature/monthly-report
```

#### ステップ8: デプロイ（Deploy）

**目的**: ユーザーが使える状態にする

```markdown
チェックリスト:
[ ] 本番環境で動作確認
[ ] エラーが発生していないか確認
[ ] パフォーマンスに問題がないか確認
[ ] ロールバック方法を理解している
```

**このプロジェクトでは**:
- GitHub Pagesに自動デプロイ（main ブランチマージ時）
- 数分で反映される
- 問題があれば元に戻せる

#### ステップ9: 監視（Monitor）

**目的**: リリース後の問題を早期発見する

```markdown
チェックリスト:
[ ] 実際に使ってみた
[ ] エラーログを確認した（ブラウザの開発者ツール）
[ ] ユーザーからのフィードバックを確認した
[ ] 必要に応じてバグ修正のIssueを作成した
```

## 3. 日常的な開発作業

### 3.1 作業開始時のルーチン（毎日）

```bash
# 1. 最新の状態に更新
git checkout main
git pull origin main

# 2. 依存関係を更新（package.jsonが更新されている場合）
npm install

# 3. テストが通ることを確認
npm test

# 4. 今日の作業ブランチを作成
git checkout -b feature/todays-work
```

**所要時間**: 3-5分

### 3.2 作業中のルーチン（随時）

```bash
# 小さな変更ごとにコミット（30分〜1時間ごと）
git add .
git commit -m "feat: add validation for date input"

# 定期的にテスト実行（変更を加えたら）
npm test

# 定期的にリント（1時間ごと、またはファイル保存時）
npm run lint:fix
```

**ポイント**: 小さく頻繁にコミットする（後で戻しやすい）

### 3.3 作業終了時のルーチン（毎日）

```bash
# 1. すべての変更をコミット
git add .
git commit -m "wip: save today's progress"

# 2. リモートにプッシュ（バックアップ）
git push origin feature/todays-work

# 3. 明日の作業をメモ（README.md や TODO.md）
echo "- [ ] 〇〇を完成させる" >> TODO.md
```

**所要時間**: 3-5分

### 3.4 週次のルーチン（毎週）

```markdown
チェックリスト:
[ ] 今週の成果を振り返る（何ができた？）
[ ] テストカバレッジを確認する
[ ] ドキュメントを更新する
[ ] 依存関係の更新を確認する（npm outdated）
[ ] 来週の目標を設定する
```

**所要時間**: 30分〜1時間

## 4. 標準チェックリスト

### 4.1 新機能追加時のチェックリスト

```markdown
## 計画
- [ ] 機能の目的を明確にした
- [ ] 既存機能への影響を確認した
- [ ] 変更範囲を最小限にした

## 実装
- [ ] 新しいブランチを作成した
- [ ] コードを小さな単位で書いた
- [ ] 既存のコードスタイルに合わせた
- [ ] 不要なコメントやデバッグコードを削除した

## テスト
- [ ] ユニットテストを書いた
- [ ] すべてのテストがパスした
- [ ] 手動でブラウザ確認した
- [ ] エッジケースを確認した

## ドキュメント
- [ ] README.mdを更新した（必要に応じて）
- [ ] JSDocコメントを追加した
- [ ] CHANGELOG.mdに記録した（リリース時）

## レビュー準備
- [ ] リントエラーがない（npm run lint）
- [ ] フォーマット済み（npm run format）
- [ ] Pull Requestの説明を書いた
- [ ] スクリーンショットを添付した（UI変更時）

## マージ後
- [ ] 本番環境で動作確認した
- [ ] 作業ブランチを削除した
```

### 4.2 バグ修正時のチェックリスト

```markdown
## 調査
- [ ] バグを再現できた
- [ ] 原因を特定した
- [ ] 影響範囲を理解した

## 修正
- [ ] 最小限の変更で修正した
- [ ] 修正のテストを追加した（再発防止）
- [ ] 既存機能を壊していない

## 検証
- [ ] バグが修正されたことを確認した
- [ ] 関連する機能も確認した
- [ ] すべてのテストがパスした

## ドキュメント
- [ ] バグの原因をコメントで説明した
- [ ] 必要に応じてドキュメント更新した
```

### 4.3 リリース時のチェックリスト

```markdown
## リリース前
- [ ] すべてのテストがパスしている
- [ ] E2Eテストがパスしている
- [ ] コードレビューが完了している
- [ ] セキュリティスキャンでアラートがない
- [ ] CHANGELOG.mdを更新した
- [ ] バージョン番号を更新した（package.json）

## リリース
- [ ] main ブランチにマージした
- [ ] GitHubでリリースタグを作成した
- [ ] デプロイが成功した

## リリース後
- [ ] 本番環境で動作確認した
- [ ] エラーログを確認した
- [ ] ユーザーに通知した（必要に応じて）
- [ ] 次のバージョンの計画を立てた
```

## 5. トラブルシューティング

### 5.1 よくある問題と解決法

#### 問題1: テストが失敗する

**症状**: `npm test` を実行するとエラーが出る

**解決手順**:
```bash
# 1. エラーメッセージをよく読む（ヒントが書いてある）

# 2. 依存関係を再インストール
rm -rf node_modules
npm install

# 3. 特定のテストのみ実行（問題を絞り込む）
npm test -- your-file.test.js

# 4. デバッグモードで実行
npm test -- --inspect-brk
```

#### 問題2: リントエラーが出る

**症状**: `npm run lint` でエラーが表示される

**解決手順**:
```bash
# 1. 自動修正を試す
npm run lint:fix

# 2. まだエラーがあれば手動修正
# （エラーメッセージの指示に従う）

# 3. どうしても解決できない場合
# ESLint設定を確認（eslint.config.js）
```

#### 問題3: Gitコンフリクトが発生

**症状**: `git merge` や `git pull` でコンフリクトが起きる

**解決手順**:
```bash
# 1. どのファイルでコンフリクトしているか確認
git status

# 2. コンフリクトファイルを開く
# <<<<<<< HEAD
# 自分の変更
# =======
# 他人の変更
# >>>>>>> branch-name

# 3. どちらを採用するか決めて編集

# 4. コンフリクト解決をコミット
git add .
git commit -m "fix: resolve merge conflict"
```

**ヒント**: コンフリクトを避けるには
- こまめに `git pull` する
- 小さな変更を頻繁にマージする
- 同じファイルを複数人で編集しない

#### 問題4: デプロイが失敗する

**症状**: GitHub Actionsでエラーが出る

**解決手順**:
```bash
# 1. GitHub Actionsのログを確認
# （GitHubのウェブサイトで確認）

# 2. ローカルで同じコマンドを実行
npm run lint
npm test
npm run e2e

# 3. エラーを修正してプッシュ
git add .
git commit -m "fix: resolve CI errors"
git push
```

#### 問題5: ブラウザで動かない

**症状**: ローカルサーバーで動作しない

**解決手順**:
```bash
# 1. ブラウザの開発者ツールでエラー確認
# （F12 キーを押す → Console タブ）

# 2. ファイルパスを確認
# （file:// プロトコルでは動かない場合がある）

# 3. ローカルサーバーを起動
./scripts/run_local.sh
# または
python3 -m http.server 8000

# 4. ブラウザで http://localhost:8000 を開く
```

### 5.2 助けを求める方法

**Issue を作成する**:

```markdown
タイトル: [Bug] 日付入力でエラーが発生する

## 環境
- OS: macOS 14.0
- ブラウザ: Chrome 120
- Node.js: v20.10.0

## 再現手順
1. index.htmlを開く
2. 日付を入力する
3. エラーが表示される

## 期待される動作
エラーなく保存される

## 実際の動作
コンソールに "TypeError: Cannot read property 'value' of null" と表示される

## スクリーンショット
（エラーのスクリーンショットを添付）

## 試したこと
- ブラウザのキャッシュをクリア
- 依存関係を再インストール
- 他のブラウザでも同じ現象
```

**ヒント**: 
- 具体的に書く（「動かない」ではなく「〇〇すると△△になる」）
- 環境情報を含める
- 再現手順を明確に
- 試したことを記載

## 6. 用語集

### 6.1 Git/GitHub関連

| 用語 | 意味 | 例 |
|-----|------|-----|
| **Repository（リポジトリ）** | プロジェクトのコード保管場所 | このプロジェクト全体 |
| **Commit（コミット）** | 変更を記録すること | `git commit -m "fix bug"` |
| **Branch（ブランチ）** | 作業の枝分かれ | feature/new-function |
| **Merge（マージ）** | ブランチを統合すること | PRをマージする |
| **Pull Request（PR）** | コードレビュー依頼 | 変更を提案する |
| **Clone（クローン）** | リポジトリをコピーすること | `git clone <URL>` |
| **Push（プッシュ）** | ローカルの変更をリモートに送る | `git push origin main` |
| **Pull（プル）** | リモートの変更をローカルに取得 | `git pull origin main` |
| **Conflict（コンフリクト）** | 変更が競合すること | 同じ行を別々に編集した |

### 6.2 開発プロセス関連

| 用語 | 意味 | 重要度 |
|-----|------|--------|
| **CI/CD** | 継続的インテグレーション/デプロイ | 🔴 高 |
| **Unit Test** | 関数単位のテスト | 🔴 高 |
| **E2E Test** | エンドツーエンドテスト（全体の動作確認） | 🟠 中 |
| **Code Review** | コードを他の人が確認すること | 🔴 高 |
| **Linter** | コードスタイルをチェックするツール | 🟠 中 |
| **Coverage** | テストがカバーしているコードの割合 | 🟡 中 |
| **Deploy** | アプリを公開すること | 🔴 高 |
| **Refactoring** | 機能を変えずにコードを改善すること | 🟡 中 |

### 6.3 このプロジェクト固有の用語

| 用語 | 意味 | ファイル |
|-----|------|---------|
| **DayLog** | 日次トレーニングログ | db.js |
| **WeekTarget** | 週間目標 | db.js |
| **ISO Week** | ISO 8601週番号 | iso-week.js |
| **Elevation Gain** | 獲得標高 | - |
